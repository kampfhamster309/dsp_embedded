# DSP Embedded – Deviation Log

All deviations from the [Dataspace Protocol specification (DSP v2024/1)](https://docs.internationaldataspaces.org/ids-knowledgebase/v/dataspace-protocol) are listed here with the spec section, rationale, and interoperability impact. No entry may have an empty rationale or impact field.

---

## Deviation Table

| ID | Spec Section | Endpoint / Feature | Deviation | Rationale | Interoperability Impact |
|----|-------------|-------------------|-----------|-----------|------------------------|
| DEV-001 | DSP v2024/1 § "Message Types" — JSON-LD `@context` | JSON-LD processing | JSON-LD contexts are statically pre-compiled at build time; no runtime JSON-LD processor is present. All namespace prefixes and message type mappings are defined as C string literals in `dsp_jsonld_ctx.h`. Remote context dereferencing, JSON-LD expansion, compaction, and framing are not performed. Additionally, incoming `TransferRequestMessage` bodies must supply `@context` as a plain string, not a JSON array, because the validator uses `cJSON_IsString()`. | An RDF/JSON-LD runtime (e.g. jsonld-java, pyLD) requires several hundred KB RAM and network access to dereference remote context documents — both infeasible on the ESP32-S3. The static table approach consumes zero additional RAM (all strings live in flash). | Counterparts must use only the standard DSP v2024/1 namespaces (`dspace:`, `dcat:`, `odrl:`, `dct:`). Custom JSON-LD extensions or non-standard context URLs will not be processed. `TransferRequestMessage` bodies must use `@context` as a plain string; sending an array causes HTTP 400. |
| DEV-002 | IDS Information Model — Identity and Trust Protocol (DAPS) | DAPS / OAuth2 | Full DAPS/OAuth2 is not implemented on the node. Identity uses pre-provisioned X.509 certificates (embedded at flash time) with local JWT validation (ES256/RS256 via mbedTLS hardware crypto). An optional DAPS gateway shim (`CONFIG_DSP_ENABLE_DAPS_SHIM=1`) proxies token requests to a configured external gateway. | DAPS requires a complex OAuth2 PKI stack (IDS CA, token endpoint, key material distribution) that is infeasible to run on a microcontroller. The X.509 approach achieves the same identity goal with zero runtime infrastructure. | Counterparts requiring DAPS-issued Dynamic Attribute Tokens (DATs) must be configured to accept X.509-based identity, or the optional DAPS gateway shim must be deployed and configured via `CONFIG_DSP_DAPS_GATEWAY_URL`. |
| DEV-003 | DSP v2024/1 § "Normative Statements" — Participant Roles | Consumer role | The node acts as data provider only. Consumer-side DSP endpoints (e.g. initiating contract negotiations, requesting catalog from another node) are not implemented. | Reduces code size and RAM footprint for the primary use case: an embedded sensor node that provides data, not consumes it. Consumer logic would add a full HTTP client stack plus state machines for the consumer protocol side. | The node cannot initiate data requests or respond to requests where it must act as consumer. All DSP interactions must be initiated by the remote counterpart (consumer). |
| DEV-004 | DSP v2024/1 § "Contract Negotiation Protocol" — Provider state transition callbacks; § "Transfer Process Protocol" — Provider state transition callbacks | Provider-to-consumer push callbacks | The node does NOT POST callback messages to the consumer's `dspace:callbackAddress` after state transitions (e.g. after agree, after transfer start or completion). The `dspace:callbackAddress` field in incoming messages is accepted and ignored. Consumers must poll state via `GET /negotiations/{id}` and `GET /transfers/{id}`. | Implementing an outbound HTTP(S) client would require a second TLS context and HTTP client stack — approximately 30–50 KB additional RAM, bringing the total DSP overhead close to the 130 KB budget ceiling. The polling approach achieves the same eventual consistency with no additional RAM. | Counterparts relying on push callbacks (blocking until a `POST /callback/negotiations/{id}/agreement` arrives) will time out. Consumers must poll `GET /negotiations/{id}` and `GET /transfers/{id}` to observe state transitions. |
| DEV-005 | DSP v2024/1 § "Contract Negotiation Protocol — Contract Negotiation States" | Simplified negotiation state machine | The full DSP negotiation state machine defines states: REQUESTED, OFFERED, ACCEPTED, AGREED, VERIFIED, FINALIZED, TERMINATED. DSP Embedded implements: INITIAL → OFFERED → AGREED, with TERMINATED optionally reachable (`CONFIG_DSP_ENABLE_NEGOTIATE_TERMINATE=1`). States REQUESTED, ACCEPTED, VERIFIED, and FINALIZED are not implemented. The `dspace:eventType` field in responses uses `dspace:OFFERED` and `dspace:AGREED`. | The provider role for a sensor node only requires: receiving an offer (→ OFFERED) and signalling acceptance (→ AGREED). The intermediate states (ACCEPTED, VERIFIED, FINALIZED) serve multi-round protocol flows and legal verification workflows that are unnecessary for sensor data transfer and would add significant state machine and handler complexity. | Counterparts expecting a FINALIZED state as the terminal agreed state must instead treat AGREED as final. Counterparts that refuse to initiate a transfer without observing FINALIZED will not interoperate without reconfiguration. The ACCEPTED state is never entered; counterparts sending a `ContractAgreementMessage` to trigger ACCEPTED will receive an unexpected response. |
| DEV-006 | DSP v2024/1 § "Transfer Process Protocol — Transfer Process States" | Simplified transfer state machine | The full DSP transfer state machine defines states: REQUESTED, STARTED, SUSPENDED, COMPLETED, TERMINATED. DSP Embedded implements: INITIAL → TRANSFERRING (maps to STARTED) → COMPLETED / FAILED. SUSPENDED and TERMINATED transfer states are not implemented. The `@type` field in transfer responses uses `dspace:TransferStartMessage` for TRANSFERRING and `dspace:TransferCompletionMessage` for COMPLETED. | Transfer suspension requires serializing in-flight transfer state and implementing a re-activation flow — significant complexity with no benefit for the continuous-poll sensor use case. The FAILED terminal state covers error conditions without needing a separate TERMINATED state. | Counterparts expecting to SUSPEND a transfer (e.g. by posting a `TransferSuspensionMessage`) will receive an error response. Counterparts must complete or abort transfers rather than suspending them. The internal state name TRANSFERRING corresponds to the spec state STARTED; message types remain DSP-compliant (`dspace:TransferStartMessage`). |
| DEV-007 | DSP v2024/1 — no concurrent session limit defined in specification | Concurrent session limits | Maximum 4 concurrent negotiations (`CONFIG_DSP_MAX_CONCURRENT_NEGOTIATIONS`, default 4) and 2 concurrent transfers (`CONFIG_DSP_MAX_CONCURRENT_TRANSFERS`, default 2). Slot tables are fixed-size and never reclaimed during a running session; once all slots are consumed, new negotiations or transfers are rejected with HTTP 500. | Fixed-size slot tables eliminate heap allocation and fragmentation on the constrained ESP32-S3. The sensor use case typically has at most 1–2 active consumers simultaneously; the defaults provide sufficient headroom while keeping the state table small enough to fit entirely in RTC slow memory for deep-sleep persistence (DSP-505). | Counterparts opening more than 4 simultaneous negotiations or more than 2 simultaneous transfers will receive HTTP 500 responses until the device is rebooted. Counterparts should not open multiple simultaneous negotiations for the same asset. Both limits are configurable at compile time via Kconfig. |
